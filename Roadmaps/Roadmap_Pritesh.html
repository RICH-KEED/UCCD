Login + JWT + Role-Based Access Control

AUTHModule 2 — JWT Authentication System

Users log in with email + password. Get back a JWT token. Every subsequent API call includes that token in the Authorization header. Role-based access: AGENT can only see their own complaints, SUPERVISOR sees everything, COMPLIANCE sees regulatory complaints.

Install
pip install python-jose[cryptography] passlib[bcrypt] python-multipart
1
Create api/models/user.py — SQLAlchemy User model: id, email, hashed_password, role (ENUM: AGENT/SUPERVISOR/COMPLIANCE/ADMIN), name, department, is_active, created_at.
2
Create users table in DB. Write Alembic migration. Seed 4 test users — one per role — with hashed passwords using passlib.hash.bcrypt.hash("password123").
3
Write api/auth.py: create_access_token(data: dict) -> str using python-jose HS256. Token expiry: 60 minutes. Secret key from config.SECRET_KEY.
4
Write verify_token(token: str) -> dict — decodes JWT, raises HTTPException 401 if invalid or expired.
5
Write get_current_user(token: str = Depends(oauth2_scheme)) -> User FastAPI dependency. Abhineet adds this to any route that needs auth.
6
Write require_role(role: str) dependency factory — raises 403 if current user's role doesn't match. Example: @router.get("/regulatory", dependencies=[Depends(require_role("COMPLIANCE"))]).
7
Add auth route to Abhineet's FastAPI: POST /api/v1/auth/login accepts email+password, returns {access_token, token_type, role, user_id, name}.
8
Test: login with test user, copy token, call protected endpoint with Authorization: Bearer {token} header. Verify 401 without token, 403 with wrong role.
Input (POST /auth/login)
  • email
  • password
  • role (which role they're logging in as)
Output
  • access_token — JWT string
  • token_type: "bearer"
  • role
  • user_id
  • name
Week 3 — Redis SLA Engine
WEEK 03
SLA Countdown Timers + Breach Detection + Alerts
Days 15–21
REDISModule 3 — SLA Engine

Uses Redis TTL keys as countdown timers for every open complaint. A background job runs every 60 seconds, checks all active SLA timers, and fires alerts when complaints reach 50%, 75%, and 90% of their deadline — before breaching. When TTL expires, marks complaint as BREACHED.

Install
pip install redis apscheduler
1
Write services/sla_service.py. Connect to Redis: r = redis.Redis(host="localhost", port=6379, decode_responses=True).
2
Write set_sla_timer(complaint_id: str, sla_tier: str) — maps tier to hours (REGULATORY=5, HIGH=24, MEDIUM=48, NORMAL=72), calls r.setex(f"sla:{complaint_id}", hours*3600, sla_tier). Also stores metadata: r.hset(f"sla_meta:{complaint_id}", mapping={"deadline": iso_string, "tier": sla_tier, "alert_50": "0", "alert_75": "0", "alert_90": "0"}).
3
Write get_sla_status(complaint_id: str) -> dict — returns time remaining in seconds, percentage elapsed, deadline ISO string. Used by Hemant's frontend countdown timers.
4
Write check_all_slas() — background job that runs every 60 seconds. Scans all sla:* keys. For each: compute pct_elapsed from TTL. If ≥ 0.50 and alert_50 not sent → call fire_sla_alert(complaint_id, "50_PERCENT") and set flag. Same for 75% and 90%.
5
Write fire_sla_alert(complaint_id, alert_type) — calls Abhineet's WebSocket broadcast: manager.broadcast({"type": "sla_alert", "alert_type": alert_type, "complaint_id": complaint_id}). Also calls supervisor_notification_service.
6
Handle breach: when r.ttl(key) returns -2 (key expired), update complaint status to "breached" in PostgreSQL and broadcast {"type": "breach_occurred", "complaint_id": ...}.
7
Write clear_sla_timer(complaint_id) — called when complaint is resolved. Deletes both the TTL key and meta hash from Redis.
8
Register the background job in Abhineet's FastAPI startup: scheduler.add_job(check_all_slas, "interval", seconds=60). Start scheduler on app startup event.
Input
  • complaint_id
  • sla_tier from Akash's severity scorer
Output
  • Redis TTL key tracking deadline
  • WebSocket alerts at 50/75/90% elapsed
  • DB status update to "breached" on expiry
Week 4 — Agent Load Tracking + Regulatory Module
WEEK 04
Cognitive Load State in Redis + Regulatory Deadlines
Days 22–28
REDISModule 4 — Agent Load Tracking

Track each agent's real-time cognitive load score in Redis. This is what powers the Agent Load Bars on the Supervisor dashboard and the routing logic in the Cognitive Load Manager.

1
Write services/agent_service.py. When a complaint is assigned to agent X, push to r.lpush(f"queue:{agent_id}", complaint_id).
2
Write compute_agent_load(agent_id: str) -> float: queue_depth = r.llen(f"queue:{agent_id}"), complexity_avg = float(r.get(f"complexity:{agent_id}") or 0), shift_hours = hours since login from Redis. Load = (queue*0.35 + complexity*0.30 + shift_fatigue*0.15 + 0.20). Clamp 0–1.
3
Write get_best_agent(complaint_severity: str, db) -> str — gets all active agents, computes load for each, returns the agent_id with lowest load who has the right skill (severity level). This is called by the orchestrator when assigning a complaint.
4
Write check_agent_loads() background job — runs every 5 minutes. Any agent with load > 0.80 → broadcast {"type": "agent_overload", "agent_id": ..., "load": ...} to WebSocket.
5
Add route to Abhineet's API: GET /api/v1/agents/load — calls compute_agent_load for all active agents, returns list for Hemant's load bars.
MODULEModule 5 — Regulatory Deadline Tracker

Regulatory complaints have an independent deadline timer separate from the internal SLA. This runs alongside the SLA engine but uses different hours and cannot be cancelled even if the complaint is "resolved".

1
Write services/regulatory_service.py. When a complaint with regulatory_flag=True arrives, set: r.setex(f"regulatory:{complaint_id}", 5*3600, "REGULATORY") (5 hours for Banking Ombudsman, 30 days for RBI formal — use 5h for MVP).
2
Write check_regulatory_deadlines() — separate background job, runs every 15 minutes. Checks all regulatory:* keys. At 80% elapsed: broadcast urgent alert. At expiry: flag complaint as REGULATORY_BREACH, notify compliance team via WebSocket.
3
Write get_regulatory_status(complaint_id) -> dict — returns time remaining, percentage, deadline. Hemant uses this for the countdown timers on the Regulatory Dashboard.
