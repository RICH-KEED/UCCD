Week 2 — Complaint DNA Fingerprinting
WEEK 02
Sentence Embeddings + Similarity Search
Days 8–14
MLModule 3 — Complaint DNA Agent

Every complaint gets a 384-dimensional vector embedding. New complaints are compared against the database to find duplicates (same customer, same issue) and clusters (different customers, same root cause). This is the Complaint DNA Fingerprinting innovation.

Install
pip install sentence-transformers torch
1
Load model: SentenceTransformer('all-MiniLM-L6-v2') — free, fast, 384 dimensions, no API key needed.
2
Write agents/dna_agent.py — generate_embedding(text: str) -> list[float] function. Returns 384-dim vector.
3
Write find_similar_complaints(embedding, complaint_type, db, threshold=0.75) — runs the pgvector cosine similarity query against the complaints table filtered by same complaint_type. Returns top 5 matches with similarity scores.
4
Write clustering logic: if similarity > 0.90 AND same customer_id → mark as duplicate. If similarity > 0.75 AND different customer → add to existing cluster or create new cluster.
5
Write DNAConsumer(BaseConsumer) — consume from complaints.translated, call embedding + similarity, write embedding and cluster_id back to the complaint row in PostgreSQL. DNAConsumer consumes from complaints.translated topic — NOT complaints.inbound.
6
Test: insert 10 complaints — 3 should be similar. Verify the cluster_id groups them correctly.
Input
  • complaint_id (UUID)
  • raw_text (string)
  • complaint_type (string)
Output
  • embedding — vector(384) stored in DB
  • cluster_id — UUID or null
  • is_duplicate — boolean
  • similar_complaints — list of IDs
Week 3 — Severity Scorer
WEEK 03
Composite Severity Score + SLA Tier Assignment
Days 15–21
MLModule 4 — Severity Scorer Agent

Computes a weighted severity score (0–100) from multiple signals. Then assigns the complaint to one of 4 SLA tiers based on that score. No ML model needed here — this is a rule-based weighted formula.

1
Write agents/severity_agent.py — compute_severity(complaint: dict) -> dict.
2
Implement scoring formula: complaint_type weight (fraud=30, regulatory=30, billing=20, service=10) + vip_customer bonus (+20) + regulatory_flag bonus (+30) + emotion_arc slope penalty (slope < -0.15 adds +15) + reopen_count penalty (each reopen adds +5, max +20).
3
Assign SLA tier: score ≥ 80 → REGULATORY (5h), score ≥ 60 → HIGH (24h), score ≥ 40 → MEDIUM (48h), else → NORMAL (72h).
4
Write SeverityConsumer(BaseConsumer) — reads from complaints.translated, computes score, writes severity_score and sla_tier back to PostgreSQL row. SeverityConsumer consumes from complaints.translated topic — NOT complaints.inbound.
5
Write unit tests for all 4 SLA tier boundaries. Test with 20 different complaint combinations.
Input
  • complaint_type
  • regulatory_flag (bool)
  • vip_customer (bool)
  • emotion_arc_slope (float)
  • reopen_count (int)
Output
  • severity_score — float 0–100
  • sla_tier — REGULATORY/HIGH/MEDIUM/NORMAL
  • sla_hours — int (5/24/48/72)
Week 4 — Emotion Arc Tracking
WEEK 04
Per-Turn Sentiment Trajectory
Days 22–28
MLModule 5 — Emotion Arc Agent

Tracks customer sentiment at every message turn — not one static score. Computes a slope (is sentiment improving or worsening?) and outputs a tone recommendation for the draft response. This is the Emotion Arc innovation.

Install
pip install transformers torch
1
Load model: pipeline("sentiment-analysis", model="cardiffnlp/twitter-roberta-base-sentiment-latest") — free HuggingFace model, works well for short messages.
2
Write agents/emotion_agent.py — compute_emotion_arc(turns: list[str]) -> dict.
3
For each turn, get sentiment label (POSITIVE/NEGATIVE/NEUTRAL) and score. Convert to numeric: POSITIVE=+score, NEGATIVE=-score, NEUTRAL=0.
4
Compute slope: (last_score - first_score) / max(len(turns)-1, 1). Negative slope = worsening. Positive = improving.
5
Set escalation_signal = True if slope < -0.15. Set recommended_tone: slope < 0 → "empathetic", regulatory_flag → "regulatory_formal", else → "professional".
6
Write EmotionConsumer(BaseConsumer) — consume from complaints.translated, then write full emotion_arc JSONB to complaint row in PostgreSQL. EmotionConsumer consumes from complaints.translated topic — NOT complaints.inbound.
7
For single-turn complaints (first message only), still run it — slope will be 0, tone will be "professional" by default.
Input
  • turns — list of message strings (customer messages only)
Output
  • arc — list of {turn, label, score}
  • arc_slope — float
  • escalation_signal — bool
  • recommended_tone — string
Week 5 — Breach Prediction Model
WEEK 05
Predictive Escalation Model
Days 29–35
MLModule 6 — Breach Prediction Agent

Predicts whether a complaint will breach its SLA deadline at the moment it is created. Uses a RandomForest classifier trained on synthetic data for MVP. This is the Predictive Escalation innovation — acts before breach, not after.

Install
pip install scikit-learn joblib numpy pandas
1
Generate synthetic training data in ml/generate_training_data.py — 5000 rows. Features: complaint_type (encoded), severity_score, queue_depth, agents_available, hour_of_day, day_of_week, historical_handle_time_p90. Label: did_breach (0 or 1). Use realistic distributions — ~25% breach rate.
2
Train in ml/train_breach_model.py: RandomForestClassifier(n_estimators=100, random_state=42). Split 80/20, print accuracy and precision/recall on test set.
3
Save model: joblib.dump(model, "ml/breach_predictor.pkl").
4
Write agents/escalation_agent.py — predict_breach(features: dict) -> dict. Loads the .pkl model, runs prediction, returns probability and pre_escalate flag (True if prob > 0.70).
5
Write EscalationConsumer(BaseConsumer) — consume from complaints.translated, query current queue depth and agents_available from Redis (Suryansh will set these up — use mock values until Week 5), run prediction, write breach_probability and pre_escalate to complaint row. EscalationConsumer consumes from complaints.translated topic — NOT complaints.inbound.
6
If pre_escalate = True, also publish a message to a new Kafka topic escalations.predicted — Suryansh's SLA engine will consume this.
Input
  • complaint_type
  • severity_score
  • queue_depth (from Redis)
  • agents_available (from Redis)
  • hour_of_day, day_of_week
Output
  • breach_probability — float 0–1
  • pre_escalate — bool
  • Event on escalations.predicted if True
